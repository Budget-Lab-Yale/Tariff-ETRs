# Statutory Rate CSV Interface: Status and Next Steps

*2026-03-26*

## What We Changed

The old interface between tariff-rate-tracker and Tariff-ETRs was a set of YAML config files generated by `generate_etrs_config.R`. This bridge was lossy: it compressed a full product×country rate matrix into headline rates + overrides, causing USMCA double-counting, modal-rate approximation errors, and missing country rates. The overall divergence was +1.40pp.

We replaced this with a dense `statutory_rates.csv.gz` — one row per product×country, one column per tariff authority. The CSV contains pre-USMCA, pre-metal-content, pre-auto-rebate statutory rates. ETRs reads it, applies all adjustments, and produces final tariff levels.

### Changes in Tariff-ETRs

- **`load_statutory_csv()`**: Reads the CSV, discovers s232 programs dynamically from `^s232_` column prefixes, extracts MFN at hs10×country level, validates against `metal_programs`.
- **`load_scenario_config()`**: Auto-detects CSV vs YAML-only mode. CSV path loads the CSV + optional YAML overlay. YAML path unchanged (backward compatible).
- **`load_yaml_overlay()` + `merge_overlay()`**: Counterfactual YAML modifications on top of CSV baseline.
- **`calc_weighted_etr()`**: New `mfn_rates_by_product_country` parameter for CSV path MFN. MFN exemption shares applied in both paths. Auto tariff name list expanded to include tracker program names.
- **Metal content**: Removed GTAP-sector fallback for unmatched products, replaced with flat 0.50 (matches tracker).

### Changes in tariff-rate-tracker

- **`export_statutory_rates()`**: New function that builds the dense CSV with per-program s232 columns, MFN, all authorities, and conditional target_total floors.
- **`generate_etrs_config()`**: Simplified to call `export_statutory_rates()` + `generate_other_params_yaml()`. Accepts `ch99_data` as parameter.
- **Statutory rate capture**: `statutory_rate_232` saved pre-rebate/pre-deal (step 4, before auto rebate and deal overrides). Other authority statutory rates saved pre-USMCA (after step 6b).
- **Metal content upgrade**: `load_metal_content()` now returns per-metal-type columns (steel_share, aluminum_share, copper_share, other_metal_share). Primary chapter override forces metal_share=1.0 for ch72/73/76. Primary chapter and heading-overlap products excluded from derivative metal scaling.
- **Stacking upgrade**: `apply_stacking_rules()` and `compute_net_authority_contributions()` use per-type active shares for nonmetal_share computation (only the specific metal type claimed by the active 232 program is excluded from IEEPA; unclaimed metal fractions are filled by IEEPA).

## Validation Results

We validated the CSV interface against the tracker's own rate calculation for **rev_10** (April 9, 2025):

- **275,878 product×country pairs matched**
- **100.0% exact match** (|diff| < 0.0001%)
- **0.00pp overall divergence**

The bugs we found and fixed along the way (each causing product-level divergences):

| Bug | Impact before fix | Root cause |
|-----|-------------------|------------|
| Auto tariff name mismatch | CA/MX autos off by ~12pp | ETRs' `auto_tariffs` list didn't include tracker program names (`autos_passenger` vs `automobiles_passenger_and_light_trucks`) |
| USMCA content share not applied | CA/MX autos over-reduced | Same name mismatch — USMCA auto content share formula never triggered |
| MFN exemption shares skipped | +0.58pp overall | CSV path wasn't loading exemption shares; tracker applies them |
| Target_total for non-binding floors | Japan +2.00pp | CSV emitted floor for all floor countries, including those at baseline (10%) below the floor (15%) |
| Ch76 products in derivative list | China -0.47pp | Ch76 (primary aluminum) products also matched derivative prefixes → incorrectly metal-scaled to near-zero |
| Heading/derivative overlap metal scaling | China ±100pp on overlap products | Products matching both heading prefixes and derivative prefixes got metal-scaled even when heading rate dominated |
| Inactive heading classification | CA/MX ±24pp on auto parts | CSV classified products into heading programs that weren't active in the revision (no Ch99 entries) |

## What We Haven't Tested

**The 100% match is only verified for rev_10 (April 2025).** This is one of 32 revisions, and it happens to be a relatively simple one:

- No active auto_parts, copper, softwood, MHD, or bus headings (all Ch99-gated and skipped)
- No Phase 2 IEEPA surcharges (only the 10% universal baseline)
- No Section 122
- No Swiss framework override
- No IEEPA invalidation (SCOTUS ruling)

Later revisions have progressively more active programs and policy changes. The fixes we made (heading gates, primary chapter exclusion, per-type metal scaling) are most likely to break on revisions where:

- **Auto parts heading becomes active** (rev_11+): The heading/derivative overlap logic gets exercised for real
- **Phase 2 surcharges** (rev_13+?): Target_total floor binding logic matters
- **Copper/MHD/wood headings activate**: New programs in the classifier
- **Swiss framework window**: Date-bounded floor override
- **IEEPA invalidation + Section 122**: Policy regime change

## What Needs To Happen Next

### 1. All-revision validation

Run the CSV export → ETRs load → compare loop for all 32 revisions. A test script exists at `tariff-rate-tracker/scripts/test_all_revisions.R` but has a blocking issue: both repos define `load_mfn_exemption_shares()` with different parameter names (`file` in ETRs vs `path` in tracker). When both are sourced into the same R session, the function name collision causes failures.

**Fix options:**
- **(a) Two-process approach**: Tracker script recalculates and saves snapshots + CSVs for all revisions. Separate ETRs script loads CSVs and validates against snapshots. No shared R session.
- **(b) Namespace the collision**: Rename one of the `load_mfn_exemption_shares()` functions to avoid the conflict.

Option (a) is cleaner and matches how the tools would actually be used in production.

### 2. Decide where CSVs live

Currently there's no established home for tracker-generated CSVs in ETRs:
- `config/test_csv_baseline/` is a throwaway test directory
- `config/baseline/` is still YAML-only
- The tracker generates CSVs into whatever output path you give it

Need to decide: do CSVs go in `config/` (tracked in git, large files), or in a `data/` directory (gitignored, regenerated on demand)?

### 3. Switch the baseline

Once the all-revision validation passes, switch `config/baseline/` from YAML-only to CSV mode. This means the tracker becomes the single source of truth for the baseline, and YAML files become counterfactual-only.

### 4. Known remaining bugs (fixed but not yet validated across revisions)

See `docs/csv-interface-bugs.md` for the three bugs found during code review:
1. s232 target_total double-prefix (floor rates ignored)
2. s232 YAML overlay cannot zero out baseline rates
3. Ch99 gating file path doesn't exist in tracker checkout
